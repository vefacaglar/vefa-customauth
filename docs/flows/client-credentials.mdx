# Client credentials flow

The **client credentials** grant (RFC 6749 §4.4) lets a confidential client obtain an access token
**on its own behalf** — there is no end user, no browser, and no login. Use it for
machine-to-machine (M2M) callers: a background service, a daemon, or one API calling another.

The issued token represents the client itself:

- The access token's `sub` is the **client id**.
- **No ID token** and **no refresh token** are issued — the client simply requests a new token
  when the old one expires.

## Requirements

A client may use this grant only when **all** of the following hold:

1. It is a **confidential** client — `TokenEndpointAuthMethod` is not `None`. The only confidential
   method in the current version is [`private_key_jwt`](/docs/flows/client-authentication#confidential-clients-private_key_jwt).
   Public (`None`) clients are rejected with `invalid_client`.
2. It has the explicit per-client opt-in `AllowClientCredentials = true`. Otherwise the request is
   rejected with `unauthorized_client`.
3. Every requested scope is in the client's `AllowedScopes` (otherwise `invalid_scope`). `scope` is
   optional; when omitted, no `scope` claim/response value is added.

> There is no shared client-secret authentication in the current version, so a client-credentials
> caller authenticates with a `private_key_jwt` client assertion — the same mechanism described in
> [Client authentication](/docs/flows/client-authentication).

## Registering an M2M client

```csharp
new CustomAuthClient
{
    ClientId = "reporting-worker",
    DisplayName = "Reporting worker",
    AllowedScopes = { "reports.read", "reports.write" },
    AllowClientCredentials = true,
    TokenEndpointAuthMethod = CustomAuthClientAuthenticationMethod.PrivateKeyJwt,
    JwksJson = "{ \"keys\": [ { \"kty\": \"RSA\", \"kid\": \"…\", \"n\": \"…\", \"e\": \"AQAB\" } ] }",
    // No RedirectUris needed — this client never uses the browser/authorize flow.
}
```

## Requesting a token

```bash
curl -X POST https://auth.example.com/connect/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=client_credentials" \
  -d "client_id=reporting-worker" \
  -d "scope=reports.read reports.write" \
  -d "client_assertion_type=urn:ietf:params:oauth:client-assertion-type:jwt-bearer" \
  -d "client_assertion=<signed JWT>"
```

The `client_assertion` is a JWT signed with the client's private key — see
[Client authentication → sending the assertion](/docs/flows/client-authentication#sending-the-assertion).

## Response

A successful response is JSON with `Cache-Control: no-store`, containing the access token only:

```json
{
  "access_token": "eyJ…",
  "token_type": "Bearer",
  "expires_in": 3600,
  "scope": "reports.read reports.write"
}
```

There is deliberately **no** `id_token` and **no** `refresh_token`. When the access token expires,
the client requests another one with the same call.

## Error responses

| Condition | HTTP | `error` |
| --- | --- | --- |
| `client_id` missing | 400 | `invalid_request` |
| Unknown client, or a public (`None`) client | 401 | `invalid_client` |
| Client assertion missing/invalid | 401 | `invalid_client` |
| `AllowClientCredentials = false` | 400 | `unauthorized_client` |
| Requested scope outside `AllowedScopes` | 400 | `invalid_scope` |

## Validating the token at a resource server

The access token is a normal signed JWT, validated **locally** against the JWKS document — exactly
like a user-issued access token. Authorize on the token's `scope` (and `client_id`) claim. See
[Token validation & revocation](/docs/flows/token-validation-and-revocation).

> Like all access tokens here, an M2M token cannot be revoked before expiry. Keep
> `AccessTokenLifetimeSeconds` short for M2M clients.
